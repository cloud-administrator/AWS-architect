以下は **2026年7月7日時点で、Claude公式ドキュメント／公式Help Centerのみ**を根拠に整理した内容です。
結論からいうと、**Admin API keyで取るものは「Activity Feed＝監査・アクティビティイベント」だけ**、**Compliance Access Keyで取るものは「Activity Feedに加えて、chat / file / project の本文・添付・メタデータを含むeDiscoveryデータ」**です。

## 1. まず全体像：2種類のキーでできることが違います

| 区分           | 使うキー                                         |                                                 主な用途 | chat / file / project本文を取得できるか |
| ------------ | -------------------------------------------- | ---------------------------------------------------: | -----------------------------: |
| 監査イベントログ取得   | **Admin API key** `sk-ant-admin01-...`       |                Compliance APIの **Activity Feed** を読む |                         **不可** |
| eDiscovery取得 | **Compliance Access Key** `sk-ant-api01-...` | Activity Feed、チャット本文、ファイル、プロジェクト、ユーザー・ロール・グループ・設定を読む |                          **可** |

公式ドキュメントでは、Compliance APIには2種類のキーがあり、**Compliance Access Keyは全エンドポイントにアクセス可能**、**Admin API keyはActivity Feedのみ**と説明されています。また、Admin API keyはCompliance APIが有効化された後に作成されたものでないと `read:compliance_activities` スコープを持たないため、古いAdmin API keyでは403になる場合があります。([Claude Platform][1])

# 2. Admin API keyで取得できるログ

## 2.1 取得対象は「Activity Feed」です

Admin API keyで取得する監査イベントログは、実装上は **Compliance APIの `GET /v1/compliance/activities`** を呼び出して取得する **Activity Feed** です。公式ドキュメントでは、Activity Feedは、組織内で発生した **認証、チャット、ファイル、プロジェクト、管理、プラットフォーム上のアクション** を記録すると説明されています。イベントは発生から約1分以内に検索可能になり、**6年間保持**されます。([Claude Platform][2])

```bash
curl --fail-with-body -sS \
  "https://api.anthropic.com/v1/compliance/activities?limit=100" \
  --header "x-api-key: $ANTHROPIC_ADMIN_KEY"
```

## 2.2 取得できるログ項目のイメージ

Activity Feedの各レコードには、たとえば以下のような情報が入ります。

| フィールド                                   | 意味                             |
| --------------------------------------- | ------------------------------ |
| `id`                                    | Activityイベントの一意ID              |
| `created_at`                            | イベント発生時刻                       |
| `organization_id` / `organization_uuid` | どの組織で起きたか                      |
| `actor`                                 | 誰／何が実行したか                      |
| `type`                                  | イベント種別。例：`claude_chat_created` |
| 追加フィールド                                 | イベント種別ごとのIDやファイル名など            |

`actor` には、ユーザー操作を示す `user_actor`、APIキーによる操作を示す `api_actor`、Admin API keyによる管理操作を示す `admin_api_key_actor`、未認証ユーザー、Anthropic側の操作、SCIM同期などの種別があります。([Claude Platform][2])

## 2.3 監査ログとして見えるイベントの例

公式Help Centerの監査ログページでは、監査ログのイベント例として、サインイン／サインアウト、SSO、Magic Link、プロジェクト作成・削除・リネーム、プロジェクト文書の追加・削除、会話の作成・削除・リネーム、ファイルアップロード、組織招待、ユーザー削除、SSO/JIT/ドメイン設定、組織データエクスポート開始・完了などが列挙されています。([Claude ヘルプセンター][3])

わかりやすく分類すると、Admin API keyで見るActivity Feedは次のような「誰が、いつ、何をしたか」の履歴です。

| 分類        | 例                                               |
| --------- | ----------------------------------------------- |
| 認証系       | SSOログイン、Google/Appleログイン、サインアウト、Magic Link、電話認証 |
| チャット系     | 会話作成、会話削除、会話名変更                                 |
| ファイル系     | ファイルアップロード                                      |
| プロジェクト系   | プロジェクト作成、削除、リネーム、公開範囲変更、ナレッジ文書追加・削除             |
| 組織管理系     | 招待送信・承認・拒否、ユーザー削除                               |
| セキュリティ設定系 | SSO接続、JIT、ドメイン検証                                |
| データアクセス系  | 組織データエクスポート開始・完了、Compliance APIアクセス             |

## 2.4 Admin API keyでは取得できないもの

Admin API keyでは、**チャット本文、プロンプト、Claudeの回答、アップロードファイル本文、プロジェクト文書本文**は取得できません。公式ドキュメントでは、Admin API keyはActivity Feedのみで、chat / file / projectエンドポイントにAdmin API keyでアクセスすると403になると説明されています。([Claude Platform][4])

また、管理画面の「Export logs」ボタンで出力できる監査ログCSVも、チャットやプロジェクトのタイトル・本文は含まず、固有IDだけが出力されます。このUIエクスポートは過去180日分で、メールのダウンロードリンクは24時間有効です。これはAPIのActivity Feedの6年保持とは別に考える必要があります。([Claude ヘルプセンター][3])

# 3. Compliance Access Keyで取得できるeDiscoveryデータ

Compliance Access Keyは、eDiscovery用途で必要になる **Activity Feed + chat / file / project content** を取得できます。公式ドキュメントでは、Compliance APIは組織のActivity Feed、ユーザー・ロール・グループ、組織設定、claude.ai組織のチャット・ファイル・プロジェクトにプログラムからアクセスできると説明されています。([Claude Platform][1])

## 3.1 必要なスコープ

| スコープ                          | できること                                   |
| ----------------------------- | --------------------------------------- |
| `read:compliance_activities`  | Activity Feedを読む                        |
| `read:compliance_user_data`   | ユーザー、チャット、メッセージ、ファイル、プロジェクト、グループメンバーを読む |
| `delete:compliance_user_data` | チャット、ファイル、プロジェクト等を削除する                  |
| `read:compliance_org_data`    | 組織メタデータ、ロール、グループ、有効な組織設定を読む             |

公式ドキュメントは、`read:compliance_user_data` を持つCompliance Access Keyは、リンクされた組織内の全チャット、ファイル、プロジェクトを読めるため、Primary Ownerが見たことのない内容にもアクセスできると注意しています。`delete:compliance_user_data` は恒久削除が可能なので、eDiscoveryの読み取り用途だけなら付与しないのが安全です。([Claude Platform][4])

## 3.2 チャットで取得できるもの

Compliance Access Keyでは、まずユーザーIDを指定してチャットのメタデータ一覧を取得し、その後、チャットIDを使ってメッセージ本文を取得します。チャット一覧には、チャットID、名前、作成日時、更新日時、削除日時、モデル、組織UUID、プロジェクトID、ユーザー情報などが含まれます。([Claude Platform][5])

メッセージ取得エンドポイントでは、各メッセージの `role`、作成時刻、テキスト本文、ユーザーが添付したファイル、Claudeが生成したファイル、Claudeが生成・更新したArtifactsの情報が取得できます。([Claude Platform][5])

つまり、eDiscoveryで必要になりやすい以下が取得対象です。

| 種類        | 取得できる内容                                         |
| --------- | ----------------------------------------------- |
| チャットメタデータ | チャットID、タイトル、作成日時、更新日時、削除日時、モデル、ユーザー、プロジェクトID    |
| ユーザー発言    | ユーザーが入力したテキスト                                   |
| Claude応答  | Claudeが返したテキスト                                  |
| 添付ファイル参照  | ユーザーがアップロードしたファイルID、ファイル名、MIMEタイプ               |
| 生成ファイル参照  | Claudeがツールで生成したPDF、CSV、スプレッドシート等のID             |
| Artifacts | Claudeが生成したコード、Markdown等のArtifact IDとversion ID |

## 3.3 ファイルで取得できるもの

ファイルは独立して全件一覧するのではなく、チャットメッセージやプロジェクト添付の中に出てくるファイルIDを使って取得します。取得対象には、ユーザーアップロードファイルのバイナリ本文、ファイルメタデータ、Claude生成ファイル、Artifactの特定バージョン本文、プロジェクト文書の本文・メタデータがあります。公式ドキュメントでは、ファイル本文取得時に元のファイル名、MIMEタイプ、MD5などのヘッダーが返ると説明されています。([Claude Platform][5])

## 3.4 プロジェクトで取得できるもの

プロジェクトについては、プロジェクトメタデータ、詳細、添付一覧、プロジェクト文書本文を取得できます。プロジェクト添付には、PDF・画像・スプレッドシートのようなバイナリの `project_file` と、常に `text/plain` として扱われる `project_doc` があり、IDの種類に応じて適切な本文取得エンドポイントを呼び分けます。([Claude Platform][5])

| 種類          | 取得できる内容                          |
| ----------- | -------------------------------- |
| プロジェクトメタデータ | プロジェクトID、作成日時、関連情報               |
| プロジェクト詳細    | プロジェクトの詳細情報                      |
| プロジェクト添付    | `project_file` または `project_doc` |
| プロジェクト文書    | plain-text文書本文                   |
| プロジェクトファイル  | PDF、画像、スプレッドシート等のバイナリファイル        |

## 3.5 ユーザー、組織、ロール、グループ、設定も取得できます

eDiscoveryでは「どのユーザーのデータを収集するか」が重要なので、Compliance APIには組織、ユーザー、ロール、グループ、グループメンバー、有効な組織設定を取得するエンドポイントもあります。公式ドキュメントでは、これらのディレクトリ系エンドポイントを、eDiscoveryのユーザーリスト作成、レポート、外部ID管理システムとの照合に使えると説明しています。([Claude Platform][6])

有効な組織設定エンドポイントでは、保持期間、コンテンツリダクション、SSO強制、IP allowlist、セッション継続時間など、実際に効いている設定を確認できます。レスポンス例には `data_retention_periods` が含まれ、チャット保持期間が日単位で表現されています。([Claude Platform][6])

# 4. 保存期間・保持期間の整理

## 4.1 一覧

| データ種類                               |              保存・保持期間 |              管理者設定で変わるか |      最小 |                    最大 |
| ----------------------------------- | -------------------: | ----------------------: | ------: | --------------------: |
| Activity Feedイベント                   |               **6年** | 公式上、顧客管理ではなくAnthropic管理 |   6年と記載 |                 6年と記載 |
| UIの「Export logs」監査ログCSV             |          **過去180日分** |          公式上、変更可能との記載なし |      不明 |              180日分と記載 |
| chat / file / project content       | 組織のClaude.ai保持ポリシー次第 |                 **変わる** | **30日** |                **不明** |
| カスタム保持未設定時のchat / project data      |            **無期限保持** |         カスタム保持を設定すると変わる | 30日から設定 | カスタム設定の上限は不明、未設定なら無期限 |
| Compliance APIでhard-deleteしたcontent |          **即時・永久削除** |             削除APIの実行者次第 |    0日相当 |                  0日相当 |

Activity Feedは6年保持で、公式の設計ガイドでも「Activity Feed records: 6 years / Controlled by Anthropic」と整理されています。一方、チャット、ファイル、プロジェクト本文は組織のClaude.ai保持ポリシーに従い、保持期間を過ぎたコンテンツやCompliance APIでhard-deleteされたコンテンツはCompliance APIから返せないと説明されています。([Claude Platform][7])

## 4.2 chat / project contentの保持ルール

Enterpriseのカスタムデータ保持では、チャットは「その会話の最後のメッセージ時刻」から保持期間が開始し、プロジェクトは「プロジェクトの最終更新時刻」から保持期間が開始します。プロジェクトの保持期間設定は、そのプロジェクト内のチャット保持設定より優先されます。公式には、最小保持期間は30日、1か月は30日として数えると説明されています。([Claude ヘルプセンター][8])

保持期間に到達すると、チャットの場合はプロジェクト内チャットを含む全チャットとArtifactsが削除され、プロジェクトの場合はプロジェクト全体、プロジェクト内チャット、Artifactsが削除されます。削除は予定日のUTC 0時に行われ、保持期間を過ぎたデータは永久削除され復元できません。([Claude ヘルプセンター][8])

## 4.3 「最大保持期間」は公式情報では不明です

公式Help Centerでは、**最小保持期間30日**、**カスタム保持未設定時は無期限保持**とは明記されています。一方で、**カスタム保持期間として設定できる最大値**は、確認した公式ドキュメント上では明記を見つけられませんでした。したがって、案件資料には次のように書くのが正確です。([Claude ヘルプセンター][8])

> カスタム保持期間の最小値は30日。最大設定値は公式ドキュメント上不明。カスタム保持を設定しない場合、データは既定で無期限保持。

# 5. 重要な注意点

## 5.1 「ログ」と「本文」は別物です

Activity Feedには「誰が、いつ、どのチャット／ファイル／プロジェクトに対して何をしたか」というイベントが残りますが、チャット本文やファイル本文を長期に取得できるかは別問題です。Activity Feedは6年保持ですが、chat / file / project contentは組織の保持ポリシーで削除されるとCompliance APIから返せません。([Claude Platform][2])

## 5.2 soft-deleteとhard-deleteの扱いが違います

ユーザーがclaude.ai上で削除したチャットは、Compliance API上では `deleted_at` が入った状態で見える場合があります。一方、Compliance API自体でhard-deleteしたチャット、ファイル、プロジェクト文書、プロジェクトは、即時かつ永久削除され、復元ウィンドウはありません。([Claude Platform][5])

## 5.3 Claude Console / Claude API workloadのプロンプト本文は対象外です

公式の設計ガイドでは、content endpointsはclaude.aiデータのみを提供し、Compliance APIは **Claude ConsoleまたはClaude API workloadのプロンプトテキストやモデル応答を含まない** と明記されています。つまり、eDiscoveryで取得できるchat / file / project contentは、主にclaude.ai側のEnterprise利用データとして理解する必要があります。([Claude Platform][7])

## 5.4 Compliance APIの利用自体も監査されます

Compliance APIへの呼び出し自体は `compliance_api_accessed` activityとしてActivity Feedに出ます。SIEMに連携する場合は、このイベントも取り込み、どのCompliance Access KeyまたはAdmin API keyがいつコンプライアンスデータにアクセスしたかを監査できるようにするのが推奨です。([Claude Platform][7])

# 6. 実装方針としてのおすすめ

Admin API key側は、`GET /v1/compliance/activities` だけを対象にして、Activity Feedを継続取得します。初回は6年分をバックフィルし、その後はカーソル方式または時間窓方式で差分取得します。公式ドキュメントでは、Activity Feedは1分以内に検索可能、ページ上限は5,000件、全Compliance APIで親組織あたり600 requests/minの制限があると説明されています。([Claude Platform][7])

Compliance Access Key側は、eDiscovery要件に合わせて `read:compliance_user_data` と必要に応じて `read:compliance_org_data` を付与します。削除が業務要件にない場合、`delete:compliance_user_data` は付与しない方が安全です。保持期間がeDiscovery要件より短い場合は、保持期限前にチャット本文・ファイル本文を外部アーカイブへエクスポートする必要があります。([Claude Platform][4])
