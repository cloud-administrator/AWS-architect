調査範囲：**Anthropic / Claude Help Center の公式情報のみ**を参照しました。確認日：**2026年7月6日**。

## 結論

**通常のメール招待でEnterprise組織に参加しただけでは、会社メールアドレスで契約している個人のPro / Maxプランは自動解約されません。**
公式FAQでは、個人Claudeを会社メールで使っている人も招待でき、参加後は「個人アカウント」と「Team / Enterprise組織アカウント」の両方を持ち、左下のアカウントメニューで切り替えられると説明されています。([Claude ヘルプセンター][1])

ただし、次の場合は個人プランが解約・終了されます。

| ケース                            |                                                      個人プランの扱い | 主な操作主体         |
| ------------------------------ | ------------------------------------------------------------: | -------------- |
| 通常のメール招待を受けてEnterpriseに参加するだけ  |                                                  **自動解約されない** | 管理者が招待、ユーザーが参加 |
| ユーザーが個人アカウントをEnterpriseへ自主移行する |             Pro / Maxは**自動キャンセル**、日割り返金あり。Apple App Store契約は別 | ユーザー           |
| Enterprise管理者が「ドメインクレーム」を実施する  | Pro / Maxは移行時または期限到来時に**自動キャンセル**、日割り返金あり。Apple App Store契約は別 | 管理者が開始、ユーザーが選択 |

## 1. メールでユーザーを招待する手順

ここでは、Claude Enterprise組織に対して、**管理者が個別にメール招待を送る方法**をまとめます。公式では、Team / Enterprise plan の「Manage members」機能として説明されています。([Claude ヘルプセンター][1])

### 1-1. 事前確認：管理者側で確認すること

**担当：管理者 / Owner / Primary Owner**

招待前に、以下を確認します。

1. **招待対象のメールアドレスが、組織の許可済みメールドメインに含まれていること**
   公式手順では、招待時に入力するメールアドレスは、組織の allowed email domains のいずれかである必要があります。([Claude ヘルプセンター][1])

2. **利用可能な座席、seat、があること**
   招待中、つまりユーザーがまだ承諾していない「Pending invitation」の状態でも、利用可能な座席をすぐに占有します。([Claude ヘルプセンター][1])

3. **座席を追加購入できる権限があるか確認すること**
   メンバー管理は Organization Admin でも可能ですが、Billing や座席購入に関係する操作は Owner / Primary Owner の権限が必要です。([Claude ヘルプセンター][1])

4. **EnterpriseでSSO / JIT / SCIMを使う場合は、招待方式との関係を確認すること**
   SSOを有効にした後は、ユーザー追加方法として「Invite only」「JIT」「SCIM directory sync」を選ぶ設計です。Invite onlyでは、SSOアクセスだけでは組織に追加されず、手動招待が必要です。JITでは初回ログイン時に自動プロビジョニング、SCIMではディレクトリ変更に応じて自動追加・削除されます。([Claude ヘルプセンター][2])

---

### 1-2. 管理者がメール招待を送る手順

**担当：管理者以上、Admin / Owner / Primary Owner**

公式手順は以下です。

1. Claudeに、Enterprise組織の管理権限を持つアカウントでログインします。

2. **Organization settings > Members** に移動します。

3. **Add member** をクリックします。

4. 招待したいユーザーのメールアドレスを入力します。
   このメールアドレスは、組織の許可済みメールドメインである必要があります。([Claude ヘルプセンター][1])

5. 適切な **seat type** を選択します。
   現在のEnterpriseプランでは「Enterprise seat」のような単一座席モデルの場合がありますが、既存契約では legacy seat type が表示される場合があります。公式では、新しいEnterpriseプランは単一のEnterprise seatを使うと説明されています。([Claude ヘルプセンター][3])

6. ユーザーの **role / permissions** を設定します。
   代表的なロールは User、Admin、Owner、Primary Owner です。Enterpriseでは Custom role も利用できます。([Claude ヘルプセンター][4])

7. **Add members** をクリックします。

8. そのユーザーにメール招待が送信されます。招待は **21日後に失効**します。失効した場合は再招待が必要です。([Claude ヘルプセンター][1])

9. 複数人をまとめて招待する場合は、**Bulk add** から、メールアドレスをカンマ区切りまたは改行区切りで入力できます。([Claude ヘルプセンター][1])

---

### 1-3. ユーザー側の作業

**担当：招待されたユーザー**

公式情報で明確に確認できるのは、管理者が「Add members」を押すとユーザーにメール招待が送られ、招待は21日で失効する、という点です。([Claude ヘルプセンター][1])

ユーザー側では、通常は招待メールを受け取り、メール内の案内に従ってClaudeにログインまたはサインアップし、Enterprise組織への参加を完了します。ただし、**個別メール招待の受信後に表示される具体的な画面名・ボタン名・メール文面は、確認した公式記事内では不明**です。

---

### 1-4. 招待の再送・取り消し

**担当：管理者以上**

保留中の招待は **Organization settings > Members > Pending** から管理できます。

* 失効した招待を再送できます。
* 保留中の招待を取り消す場合は、Pendingタブで対象メンバーを探し、**Remove from team** を選択します。([Claude ヘルプセンター][1])

---

### 1-5. 一般メンバーからの招待について

**担当：管理者が設定、一般メンバーが招待操作**

公式情報では、組織メンバーもチームメイトをメールで招待できる機能があります。ただしEnterpriseでは、この **member-to-member invites はデフォルトで無効**です。管理者が有効化できます。([Claude ヘルプセンター][1])

管理者の設定により、挙動が変わります。

* **Approve one-by-one、デフォルト**：メンバーが招待リクエストを出し、管理者が承認した後に招待メールが送信されます。
* **Approve automatically**：招待がすぐ送信され、招待された人はすぐ参加できます。
* 管理者が無効化している場合、招待できるのは管理者のみです。([Claude ヘルプセンター][1])

---

## 2. 会社メールで個人プランを契約しているユーザーの扱い

ここが特に重要です。

## 2-1. 通常のメール招待だけでは、個人プランは解約されない

**担当：管理者が招待、ユーザーが参加**

会社メールアドレスでClaudeの個人アカウント、Free / Pro / Max、を使っている人を、Enterprise組織に招待することは可能です。公式FAQでは、参加後は「個人アカウント」と「Team / Enterprise plan account」の両方を持ち、左下のアカウントメニューから切り替えられると説明されています。([Claude ヘルプセンター][1])

したがって、通常のメール招待を受けてEnterprise組織に参加しただけでは、個人のPro / Max契約は自動解約されるとは読み取れません。むしろ、公式説明上は両方のアカウントが併存します。

### 実務上の案内文

会社メールアドレスでClaude Pro / Maxなどの個人プランを契約している場合でも、Enterprise組織への通常招待を承諾しただけでは、個人プランは自動解約されません。参加後は、同じメールアドレスに紐づく個人アカウントとEnterprise組織アカウントが併存し、Claude画面左下のアカウントメニューから切り替えて利用できます。

個人プランを終了したい場合は、個人アカウントをEnterpriseに移行する、または個人プランを別途解約する必要があります。

---

## 2-2. ユーザーが自主的にEnterpriseへ移行する場合

**担当：ユーザー**

会社メールの個人アカウントを持つユーザーは、Team / Enterprise組織へ個人アカウントを移行できる場合があります。公式では「Voluntary migration」と説明されています。([Claude ヘルプセンター][5])

選択肢は大きく2つです。

### 選択肢A：両方のアカウントを維持する

**担当：ユーザー**

ユーザーが「Keep both accounts」を選ぶ場合、個人アカウントはアクティブなまま残り、組織アカウントと切り替えて利用できます。([Claude ヘルプセンター][5])

この場合、Pro / Max契約は自動解約されません。

### 選択肢B：組織アカウントだけを使う

**担当：ユーザー**

「Use your organization account only」を選ぶと、さらに次の2つがあります。

1. **Bring your data with you**
   個人アカウントのチャット、プロジェクト、ファイル、メモリ、コネクタ、スキルを組織ワークスペースへ移動します。個人アカウントは閉じられ、Pro / Maxの有効なサブスクリプションは自動キャンセルされ、日割り返金されます。([Claude ヘルプセンター][5])

2. **Delete your data**
   個人アカウントを閉じますが、個人データは組織に移動しません。有料サブスクリプションがある場合は日割り返金されます。([Claude ヘルプセンター][5])

Enterpriseの場合、参加時にこの移行プロンプトは表示されず、ユーザーが **Settings > Account > Close your personal account** から開始すると説明されています。([Claude ヘルプセンター][5])

---

## 2-3. Apple App Store経由のPro / Max契約は自動解約されない

**担当：ユーザー**

Apple App Store経由でClaude Pro / Maxを契約している場合、Anthropic側では自動キャンセルできません。公式では、Apple App Store契約の場合はApple ID設定からユーザー自身がキャンセルする必要があり、キャンセルしないとAppleから課金が続くと説明されています。([Claude ヘルプセンター][5])

### 実務上の注意

Apple App Store経由でClaude Pro / Maxを契約しているユーザーは、Enterprise参加やアカウント移行だけではApple側の課金が止まりません。必ずユーザー本人がApple IDの「サブスクリプション」からClaudeを解約してください。

---

## 2-4. 管理者がドメインクレームを使う場合

**担当：管理者が開始、ユーザーが選択**

Enterpriseには、管理者が会社の確認済みドメイン上にある既存の個人Claudeアカウント、Free / Pro / Max、を発見し、Enterpriseワークスペースへ移行させる **domain claiming** という機能があります。これはEnterprise限定機能です。([Claude ヘルプセンター][6])

### 管理者側の前提条件

ドメインクレームを有効化するには、以下が必要です。

* 組織作成制限を有効化すること
* DNSによるドメイン検証を完了すること
* SSOを設定するだけでなく、実際に強制すること
* JITまたはSCIMを有効化すること

これらは、移行後にユーザーがClaudeへサインインできなくなることを防ぐための前提条件として説明されています。([Claude ヘルプセンター][6])

### ユーザーに起きること

管理者がドメインクレームを開始すると、対象ユーザーにはメールとアプリ内通知が届きます。期限は、管理者がクレームを開始した日から少なくとも30日後です。ユーザーは期限までに次のどちらかを選びます。([Claude ヘルプセンター][7])

| ユーザーの選択        | 内容                                                |
| -------------- | ------------------------------------------------- |
| Merge and join | 個人アカウントの会話、プロジェクト、ファイル、メモリなどをEnterpriseアカウントへ移行する |
| Join fresh     | 個人データを移行せず、クリーンなEnterpriseアカウントで開始する              |

ドメインクレーム後は、会社ドメイン上の個人アカウントとEnterpriseワークスペースは併存できません。ユーザーは merge か join fresh を選ぶ必要があります。([Claude ヘルプセンター][7])

### 個人プランの扱い

Pro / Maxサブスクリプションがある場合、ユーザーが移行した時点、または期限が到来した時点で自動キャンセルされ、未使用期間・使用クレジットについて日割り返金されます。([Claude ヘルプセンター][7])

ただし、Apple App Store経由の契約はAnthropicが直接キャンセル・返金できないため、ユーザーがApple ID設定から自分で解約する必要があります。([Claude ヘルプセンター][7])

JITは、正式には **Just-in-Time provisioning、ジャストインタイム・プロビジョニング** といいます。
Claude Enterpriseでは、**ユーザーがSSOで初めてログインしたタイミングで、Claude組織のメンバーとして自動追加する仕組み**です。公式ヘルプでは、AnthropicのIdPアプリに割り当てられたユーザーが、初回ログイン時に自動プロビジョニングされると説明されています。([Claude ヘルプセンター][1])

## まず、JITを一言でいうと

**管理者が1人ずつClaudeに招待メールを送らなくても、会社のSSOに割り当てられているユーザーなら、初回ログイン時に自動でClaude Enterpriseに参加できる仕組み**です。

たとえば、会社が Microsoft Entra ID、Okta、Google Workspace などを使っている場合、管理者がそのIdP側で「この人はClaudeアプリを使える」と割り当てておきます。
そのユーザーがClaudeで **Continue with SSO** からログインすると、その瞬間にClaude側のEnterprise組織メンバーとして作成されます。

| **ユーザー** | 招待メールを待つのではなく、ClaudeにSSOでログインする。初回ログイン時にEnterprise組織へ参加する。                |

## JITの流れ

### 1. 管理者がSSOを設定する

JITは、SSOが前提です。Claudeの公式ヘルプでも、JIT / SCIMの設定ガイドは、ドメイン検証とIdPによるSSO設定が完了していることを前提にしています。([Claude ヘルプセンター][1])

つまり、JITだけを単独で使うのではなく、順番としては以下です。

1. 会社ドメインをClaude側で検証する
2. Okta、Microsoft Entra ID、Google WorkspaceなどのIdPでSSOを設定する
3. Claude側でプロビジョニング方式としてJITを選ぶ
4. IdP側で、Claudeを使えるユーザーを割り当てる

---

### 2. 管理者がIdP側でユーザーをClaudeアプリに割り当てる

ここが重要です。

JITでは、Claudeの管理画面で1人ずつ招待するのではなく、**IdP側で「このユーザーはClaudeアプリを利用できる」と割り当てる**ことが入口になります。

例：

* Microsoft Entra IDでClaudeアプリにユーザーを割り当てる
* OktaでClaudeアプリにユーザーまたはグループを割り当てる
* Google Workspace側でClaude SSOアプリの対象ユーザーを設定する

公式ヘルプでは、JITを選ぶ場合、保存前にすべての対象ユーザーがIdPのAnthropicアプリケーションに割り当てられていることを確認するよう注意されています。([Claude ヘルプセンター][1])

---

### 3. ユーザーがClaudeにSSOで初回ログインする

**担当：ユーザー**

ユーザーはClaudeにアクセスし、会社のSSOでログインします。
この初回ログインのタイミングで、Claude Enterprise組織のメンバーとして自動作成されます。

公式には、JITでは「IdPアプリに割り当てられたユーザーはログイン時にUserロールでプロビジョニングされる」と説明されています。([Claude ヘルプセンター][1])

---

### 4. Claude側のメンバーとして作成される

JITで作成されたユーザーは、通常、初期状態では **Userロール** になります。
ロールやシートタイプは、管理者が手動変更できます。([Claude ヘルプセンター][1])

ただし、グループマッピングを使うと、IdPのグループに応じてロールやシートタイプを自動で割り当てられます。たとえば、IdPで「Claude-Admins」グループに入っている人をClaudeのAdminにする、といった設定です。公式ヘルプでも、JITとSCIMはいずれもグループマッピングと組み合わせて、IdPグループに基づくロールまたはシートティア割り当てを制御できると説明されています。([Claude ヘルプセンター][1])

| 大人数導入       | 手間がかかりやすい            | 展開しやすい                 |
| 権限の初期値      | 管理者が招待時に指定           | 基本はUser。グループマッピングで制御可能 |
| 退職者・異動者の削除  | Claude側で手動管理         | JITだけではClaude側の削除は基本手動 |

---

## SCIMとの違い

JITとSCIMは混同されやすいです。

### JIT

JITは、**ログインした瞬間にユーザーを作る方式**です。
そのため、IdPでClaudeアプリに割り当てられていても、ユーザーがまだClaudeにログインしていなければ、Claude側のメンバーとしてはまだ作成されません。

また、IdPアプリからユーザーを外すと、そのユーザーはログインできなくなりますが、Claude側のユーザーリストには、ログイン試行または手動削除まで残ると公式ヘルプに記載されています。([Claude ヘルプセンター][1])

### SCIM

SCIMは、**IdPのディレクトリ情報をClaudeに同期して、ログイン前でもユーザーを追加・削除する方式**です。
公式ヘルプでは、SCIMはIdPの割り当てに基づいて自動的にプロビジョニングおよびデプロビジョニングされ、事前ログインは不要と説明されています。([Claude ヘルプセンター][1])

### ざっくり比較

| 項目                       | JIT                 | SCIM               |
| ------------------------ | ------------------- | ------------------ |
| ユーザー作成タイミング              | 初回ログイン時             | IdP同期時             |
| ログイン前にClaude側へユーザー作成されるか | されない                | される                |
| IdPから外したときのClaude側削除     | 基本は手動削除が必要          | 自動削除               |
| 設定の簡単さ                   | 比較的シンプル             | より高度               |
| 向いているケース                 | まずSSO連携で簡単に自動参加させたい | 入退社・異動まで厳密に自動管理したい |

---

## JITを使うときの注意点

### 1. 座席数が足りないとユーザー追加できない

JITであっても、Claude Enterprise側に利用可能なseatが必要です。公式ヘルプでは、OwnerまたはPrimary Ownerが新規ユーザーをプロビジョニングできるよう、利用可能なseatを確保する必要があると説明されています。([Claude ヘルプセンター][2])

つまり、IdP側ではClaudeアプリに割り当て済みでも、Claude側の座席が不足していると、ユーザーがスムーズに参加できない可能性があります。

---

### 2. JITだけでは退職者のClaude側メンバー削除は完全自動ではない

JITでは、IdP側でClaudeアプリから外されたユーザーはログインできなくなります。
ただし、Claude側のユーザーリストからは、ログイン試行または管理者の手動削除まで残ると公式ヘルプに記載されています。([Claude ヘルプセンター][1])

厳密な入退社管理をしたい場合は、SCIMの方が適しています。

---

### 3. グループマッピングを使わない場合、基本はUserロールになる

JITでは、初回ログイン時のロールは基本的に **User** です。
Admin、Owner、Custom Roleなどを自動で割り当てたい場合は、IdPグループとClaudeのロールを対応させる **グループマッピング** を検討します。([Claude ヘルプセンター][1])

---

### 4. SSO有効化後はログイン方法が変わる

SSOを要求する設定にすると、対象ユーザーはClaude / Consoleアカウントへログインする際に **Continue with SSO** を使う必要があります。SSOを要求しない場合は、SSOログインとメールログインの選択肢が残ります。([Claude ヘルプセンター][3])

---

## 導入案件での説明例

社内向けには、次のように説明すると伝わりやすいです。

> JITは、Claude Enterpriseにユーザーを1人ずつ招待する代わりに、会社のSSOに登録されているユーザーが初めてClaudeへログインしたタイミングで、自動的にEnterprise組織へ追加される仕組みです。管理者はIdP側でClaudeを使えるユーザーを割り当て、ユーザーはClaudeにSSOでログインします。大人数展開には便利ですが、退職者のClaude側削除まで完全自動化したい場合はSCIMの検討が必要です。

## 個人プランとの関係

JITは、あくまで **Enterprise組織へのユーザー追加、プロビジョニング、の仕組み**です。
**個人プランの解約や、個人アカウントのデータ移行を行う機能ではありません。**

公式のJIT説明では、JITによって個人のPro / Maxプランが自動解約されるとは確認できませんでした。
個人プランの扱いは、前回整理したとおり、通常参加・自主移行・ドメインクレームで分けて確認する必要があります。

## 必須条件の全体像

Claude Enterpriseでドメインクレームを有効化するには、公式上、以下すべてが必要です。

| 必須条件              | 内容                                 | 主担当      |
| ----------------- | ---------------------------------- | -------- |
| 1. 組織作成制限         | 検証済みドメインで新しい個人/組織アカウントが作られないよう制限する | 管理者      |
| 2. DNSによるドメイン検証   | 会社がそのメールドメインを所有していることを証明する         | 管理者 / IT |
| 3. SSOの強制         | SSOを「設定する」だけでなく、実際に必須化する           | 管理者 / IT |
| 4. JITまたはSCIMの有効化 | ユーザーがEnterprise組織に入れる経路を用意する       | 管理者 / IT |

Anthropic公式ヘルプでは、これらはユーザーが移行中にClaudeへサインインできなくなることを防ぐための前提条件だと説明されています。([Claude ヘルプセンター][1])

## なぜJITまたはSCIMが必要なのか

ドメインクレームを行うと、会社ドメインの個人ClaudeアカウントがEnterprise側へ移行対象になります。
このときSSOが強制されているため、ユーザーは会社のIdP、たとえばOkta、Microsoft Entra ID、Google Workspaceなどを経由してClaudeに入る必要があります。

ここでJITまたはSCIMがないと、SSO認証は通っても、Claude Enterprise組織側にそのユーザーを作成・参加させる仕組みがありません。
そのため、公式では **SSO強制 + provisioning有効化** によって、移行後のユーザーがサインインできる経路を確保すると説明しています。([Claude ヘルプセンター][1])

## JITとSCIMのどちらを選べばよいか

### JITでよいケース

JITは、**ユーザーが初めてSSOログインしたタイミングでClaude Enterpriseに自動追加される方式**です。公式では、IdPアプリに割り当てられたユーザーが初回ログイン時にUserロールでプロビジョニングされると説明されています。([Claude ヘルプセンター][2])

大まかには、以下のような場合に向いています。

| 向いているケース          | 理由                                     |
| ----------------- | -------------------------------------- |
| まずはシンプルに始めたい      | SCIM連携より構成が軽い                          |
| ユーザー追加は初回ログイン時でよい | ログインするまでClaude側には作られない                 |
| 退職者削除を手動運用できる     | JITではIdPから外してもClaude側のユーザーリストには残る場合がある |

注意点として、JITではIdPアプリからユーザーを外すとログインできなくなりますが、Claude側のユーザー一覧には、ログイン試行または手動削除まで残ると説明されています。([Claude ヘルプセンター][2])

### SCIMがよいケース

SCIMは、**IdPの割り当て情報に基づいて、Claude側のユーザーを自動で追加・削除する方式**です。公式では、SCIMはユーザーがログインする前でも自動プロビジョニング・デプロビジョニングできると説明されています。([Claude ヘルプセンター][2])

大まかには、以下のような場合に向いています。

| 向いているケース          | 理由                      |
| ----------------- | ----------------------- |
| 入退社・異動を厳密に自動管理したい | IdPから外すとClaude側も自動削除される |
| 大規模展開             | ユーザーライフサイクル管理がしやすい      |
| 監査・統制を重視          | IdPを正とした管理に寄せやすい        |

## JITとSCIMを同時に有効化するべきか

**同時有効化は避けるべきです。**
公式ヘルプでは、SCIMとJITを同時に有効化しないよう記載されており、両方が有効な場合はSCIMが優先されると説明されています。([Claude ヘルプセンター][1])

したがって設計上は、次のどちらかに決めるのが安全です。

| 選択   | 推奨される状況                  |
| ---- | ------------------------ |
| JIT  | まず簡単にEnterprise参加経路を作りたい |
| SCIM | 入退社・異動まで自動管理したい          |

## ドメインクレーム前の重要チェック

ドメインクレームを開始する前に、**対象ユーザーがIdP側でClaudeアプリに割り当てられているか**を必ず確認してください。公式ヘルプでは、IdPに追加されていないユーザーは移行後にサインインできなくなる可能性があるため、クレーム有効化前に必要なユーザーをIdPへ追加するよう説明されています。([Claude ヘルプセンター][1])

特に注意すべきなのは、以下のようなユーザーです。

| ユーザー例                                   | リスク                     |
| --------------------------------------- | ----------------------- |
| 会社メールで個人Claudeを使っているが、IdPのClaudeアプリに未割当 | 移行後にEnterpriseへ入れない     |
| 退職済み・休職中・外部委託など、IdP上の扱いが曖昧              | ドメインクレーム対象になる可能性がある     |
| 別部署でClaude利用予定だが、IdPグループに入っていない         | JIT / SCIMでプロビジョニングされない |

## 実務上の結論

今回のClaude Enterprise導入案件では、次の理解で進めるのが正確です。

**ドメインクレームを使うなら、JITまたはSCIMは必須です。**
**SSOを設定しただけでは不十分で、SSOを強制し、さらにJITまたはSCIMでユーザーをEnterprise組織に参加させる仕組みを有効化する必要があります。**
小〜中規模でまず始めるならJIT、大規模・厳格なID管理ならSCIMを推奨します。
