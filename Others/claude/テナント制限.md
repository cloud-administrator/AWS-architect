# テナント制限

> GitHub表示用にDOCXからMarkdown化しています。

調査範囲：Anthropic公式の「Anthropicヘルプセンター／Claude公式ヘルプ」に限定しました。以下は Claude Enterpriseの「テナント制限 / Tenant Restrictions」 の設定手順です。

## 結論

Claude Enterpriseのテナント制御は、公式には 「テナント制限」 と説明されており、Claude管理画面でスイッチを入れる機能ではなく、社内ネットワークプロキシ／Secure Web GatewayでClaude向け通信にHTTPヘッダーを挿入する方式です。Anthropicはそのヘッダーを検証し、許可リストに含まれない組織アカウントへのアクセスをブロックします。(Claudeヘルプセンター)

## 1. 事前に理解すべきこと

テナント制限は、EnterpriseプランおよびConsoleオーガニゼーションのメンバーが利用できます。目的は、企業ネットワーク上のユーザーが、承認済みのClaude組織アカウントだけにアクセスできるようにし、個人アカウント利用を防ぐことです。(Claudeヘルプセンター)

対象として明記されている認証方式は、Webアクセス、デスクトップ／アプリアクセス、APIキー認証、OAuthトークン認証です。(Claudeヘルプセンター)

重要なのは、この制御は「社内プロキシを通過する通信」に効くという点です。公式記事でも、テナント制限が設定されていないネットワークには影響せず、管理されていないネットワークでは標準認証が継続されるとされています。(Claudeヘルプセンター)

## 2. 設定手順

### 手順1：許可するClaude組織のOrganization UUIDを確認する

Enterpriseプランでは、Organization IDを次のいずれかで確認します。

設定 > アカウント に移動し、Organization IDを確認する。

オーガニゼーション設定 > オーガニゼーション に移動し、ページ下部のOrganization IDを確認する。

Consoleオーガニゼーションの場合は、設定 > オーガニゼーション で確認します。(Claudeヘルプセンター)

### 手順2：社内プロキシでヘッダー挿入ルールを作成する

Claude向け通信に、次のヘッダーを挿入します。

```text
anthropic-allowed-org-ids: <org-uuid-1>,<org-uuid-2>,...
```

公式仕様では、Organization UUIDはカンマ区切りで、値の間にスペースを入れません。(Claudeヘルプセンター)

プロキシ側の設定値は、公式記事では次のように示されています。

| 項目 | 設定内容 |
| --- | --- |
| ルール名 | Claude テナント制限 |
| 対象アプリケーション／ドメイン | claude.ai、api.anthropic.com、claude.com、anthropic.com |
| アクション | ヘッダーを挿入 |
| ヘッダー名 | anthropic-allowed-org-ids |
| ヘッダー値 | 許可するOrganization UUID |
| TLS検査 | 必須 |

この設定はTLS検査が必須とされています。(Claudeヘルプセンター)

### 手順3：単一組織か複数組織かを決める

単一のClaude Enterprise組織だけを許可する場合は、ヘッダー値をその組織UUIDのみにします。

```text
anthropic-allowed-org-ids: <your-org-uuid>
```

複数の組織、たとえば関連会社やパートナー組織も許可する場合は、カンマ区切りで複数UUIDを指定します。

```text
anthropic-allowed-org-ids: <org-uuid-1>,<org-uuid-2>
```

公式記事でも、単一オーガニゼーションと複数オーガニゼーションの両方がユースケースとして記載されています。(Claudeヘルプセンター)

### 手順4：制限ネットワークからAPIで疎通テストする

公式記事では、制限されたネットワークから組織のAPIキーで次のようにテストするとされています。(Claudeヘルプセンター)

```text
curl https://api.anthropic.com/v1/messages \
-H "x-api-key: $CLAUDE_API_KEY" \
-H "anthropic-version: 2023-06-01" \
-H "content-type: application/json" \
-d '{"model":"claude-sonnet-4-6","max_tokens":1024,"messages":[{"role":"user","content":"Hello"}]}'
```

加えて、実務上は次の2つを確認すると安全です。

| テスト | 期待結果 |
| --- | --- |
| 許可済みOrganizationのClaudeアカウント／APIキーでアクセス | 成功する |
| 個人アカウント、または許可リスト外のOrganizationでアクセス | ブロックされる |

ブロック時のエラーは、公式記事では permission_error、tenant_restriction_violation と示されています。(Claudeヘルプセンター)

## 3. 対応プロキシ製品

公式記事でサポート対象として挙げられているのは、Zscaler ZIA、Palo Alto Prisma Access、Cato Networks、Netskope、およびヘッダー挿入機能を持つ汎用HTTPSプロキシです。(Claudeヘルプセンター)

ただし、各プロキシ製品ごとの詳細なクリック手順やポリシー画面の具体名までは、Anthropic公式記事には記載されていません。そのため、ZscalerやNetskopeなどの個別UI手順については、Anthropic公式情報だけでは不明です。

## 4. 併せて検討すべき関連設定

テナント制限はネットワークレベルの制御です。企業導入では、通常これだけでなく、SSO／JIT／SCIMやドメイン関連の制御も組み合わせて設計します。

ClaudeのSSOでは、SSO設定前にドメイン検証が必要で、Enterpriseプランでは親組織が組織設定時に自動作成されると説明されています。(Claudeヘルプセンター)

SSOを必須にすると、ユーザーはClaude／Consoleにログインする際に「SSOで続行」を使う必要があります。(Claudeヘルプセンター)

EnterpriseではSCIMプロビジョニングも利用でき、SCIMではIdPの割り当てに基づいてユーザーが自動プロビジョニング／デプロビジョニングされます。(Claudeヘルプセンター)

既存の個人Claudeアカウントを企業ドメインから整理したい場合は、Enterpriseの「ドメイン請求／ドメインキャプチャ」が別機能として用意されています。ただし、前提として、組織作成の制限、DNSによるドメイン検証、SSOの強制、JITまたはSCIMの有効化が必要です。(Claudeヘルプセンター)

## 5. 導入時の注意点

テナント制限を有効化していないネットワークには影響せず、管理されていないネットワークでは標準認証が継続されます。したがって、社外回線やプロキシを経由しない通信まで制御したい場合、テナント制限単体では不足する可能性があります。(Claudeヘルプセンター)

IPアドレスベースでも制限したい場合、Claude EnterpriseにはIP許可リスト機能があります。これは認証済みリクエストの送信元IPを組織の許可リストと照合し、許可リスト外のIPをブロックする機能です。ただし、有効化にはCIDR範囲を整理し、Anthropic担当者または営業チームに共有する必要があるとされています。(Claudeヘルプセンター)

## 6. 公式情報から不明な点

Anthropic公式記事だけでは、以下は不明です。

| 項目 | 状態 |
| --- | --- |
| 各プロキシ製品ごとの詳細な画面操作 | 不明 |
| モバイルアプリ単位での具体的な挙動 | 「デスクトップ/アプリアクセス」とは記載あり。ただしモバイル個別の明記は確認できませんでした。 |
| ブロックされたアクセス試行をどの管理画面・ログ項目で確認するか | 「監査証跡」という利点は記載されていますが、具体的な確認画面は不明 |
| Claude管理画面上でのTenant Restrictions専用トグルの有無 | 公式記事からは、プロキシでヘッダーを挿入する方式と読み取れます。専用トグルの記載は確認できませんでした。 |

まとめると、Claude Enterpriseのテナント制御は、許可するOrganization UUIDを取得し、社内プロキシで anthropic-allowed-org-ids ヘッダーをClaude向け通信に挿入し、TLS検査を有効化してテストする、という流れです。導入案件では、SSO強制・JIT/SCIM・ドメイン請求・IP許可リストとの役割分担を設計してから本番適用するのが安全です。
