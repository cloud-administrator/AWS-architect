# Claude Enterprise／Claude.aiの全体構造

**調査基準日：2026年7月23日**
AnthropicおよびClaudeの公式サイト・公式ヘルプ・公式ドキュメントのみを確認しました。

## まず結論

Claude Enterpriseは、単なる「企業向けチャット画面」ではありません。

次の要素をまとめた、**企業向けAI業務プラットフォーム**です。

1. 従業員がClaudeを使うアプリ群
2. 企業のユーザー、組織、権限を管理する仕組み
3. 社内文書や業務システムと接続する仕組み
4. チャット、ファイル、プロジェクトなどのデータ領域
5. セキュリティ、監査、コンプライアンス機能
6. 利用量、費用、導入状況を管理する機能

現在のEnterpriseプランには、Chat、Claude Code、Claude Cowork、コネクター、SSO、SCIM、監査ログなどが含まれます。 ([Anthropic][1])

# 1. 全体像

論理的な構造は、次のように考えると分かりやすいです。

```text
┌──────────────────────────────────────────────┐
│ 企業の既存環境                               │
│                                              │
│ ・IdP：Microsoft Entra ID、Okta等             │
│ ・社内ネットワーク、VPN、Proxy               │
│ ・Slack、Google Workspace、Microsoft 365等    │
└──────────────────────────────────────────────┘
                 │
                 │ SSO／SCIM／ネットワーク制御
                 ▼
┌──────────────────────────────────────────────┐
│ Parent Organization（親組織）                │
│                                              │
│ ・会社ドメインの確認                         │
│ ・SSO設定                                    │
│ ・ユーザープロビジョニング                   │
│ ・複数組織に共通するID管理                   │
└──────────────────────────────────────────────┘
                 │
        ┌────────┴────────┐
        ▼                 ▼
┌──────────────────┐  ┌────────────────────────┐
│ Claude.ai組織    │  │ Claude Console組織     │
│                  │  │                        │
│ 従業員が利用     │  │ API開発者が利用        │
│                  │  │                        │
│ ・Chat            │  │ ・APIキー              │
│ ・Claude Code     │  │ ・Workspace            │
│ ・Cowork          │  │ ・APIコスト管理        │
│ ・Projects        │  │ ・Workbench            │
│ ・Files           │  │                        │
│ ・Connectors      │  │ ※Claude Enterpriseとは│
│ ・Skills          │  │   別の製品領域         │
└──────────────────┘  └────────────────────────┘
        │
        ├─ Organization settings
        │   ├─ Members
        │   ├─ Groups
        │   ├─ Roles
        │   ├─ Models
        │   ├─ Connectors
        │   ├─ Data and privacy
        │   ├─ Billing／Usage
        │   └─ Analytics
        │
        ├─ 監査ログ／Compliance API → SIEM・DLP
        └─ Analytics API → BI・導入状況分析
```

ここで最も重要なのは、**「親組織」と「Claude.ai組織」は役割が異なる**という点です。

* 親組織：会社全体の認証・ID管理をまとめる
* Claude.ai組織：従業員、チャット、ファイル、プロジェクト、利用設定などを管理する
* Claude Console組織：Claude APIを利用する開発者向けの別領域

Anthropicの公式ドキュメントでは、親組織に複数のClaude.ai組織やClaude Console組織を関連付け、ドメイン確認、SSO、プロビジョニング設定を共有できると説明されています。一方、請求、利用量、データは各組織単位で管理されます。 ([Claudeヘルプセンター][2])

# 2. Parent Organization（親組織）

## 親組織とは

親組織は、Claudeのチャットデータを置く場所というより、**企業全体のID・認証を統括する論理的な入れ物**です。

Enterprise組織を作成すると、親組織は自動的に作成されます。親組織では、主に次の情報を管理します。

* 会社のメールドメイン
* ドメイン所有権の確認
* SSO設定
* 接続するIdentity Provider
* JIT／SCIMプロビジョニング
* IdPグループとClaude組織の対応
* 親組織に関連付けるClaude.ai組織
* 親組織に関連付けるClaude Console組織

1つの親組織に設定できるIdPは原則1つです。同じ親組織に属する複数の組織は、そのIdPおよびSSO設定を共有します。 ([Claudeヘルプセンター][2])

## 例

```text
ABC株式会社 Parent Organization
│
├─ Claude.ai組織：ABC本社
├─ Claude.ai組織：ABC研究開発部門
├─ Claude.ai組織：ABC海外部門
└─ Claude Console組織：ABC API開発基盤
```

このように分けることで、認証は共通化しつつ、組織ごとにデータ、利用設定、請求、権限を分離できます。

ただし、実際に複数組織へ分割すべきか、1つの組織内でグループとRBACを使うべきかは、次の要件によって変わります。

* データを完全に分離する必要があるか
* 子会社・海外拠点で契約や請求を分けるか
* 使用するIdPが異なるか
* 管理者を分ける必要があるか
* 監査・法務上の境界を分けるか

# 3. Claude.ai組織

Claude.ai組織は、Claude Enterpriseの中心となる**企業テナント**です。

組織の中には、次の情報が含まれます。

```text
Claude.ai組織
├─ メンバー
├─ グループ
├─ ロール・権限
├─ 契約シート
├─ 利用上限・予算
├─ チャット
├─ プロジェクト
├─ ファイル
├─ Artifacts
├─ Skills
├─ Connectors
├─ 組織設定
├─ 監査ログ
└─ 利用分析
```

ユーザーがClaude Enterpriseで作成したチャット、アップロードしたファイル、プロジェクトなどは、基本的に所属するClaude.ai組織の管理対象になります。Primary Ownerは、組織データのエクスポートを通じて、チャット、ファイル、利用状況等へのアクセスを要求できます。 ([Claudeヘルプセンター][3])

# 4. 利用者が触るアプリケーション層

Claude Enterpriseには、性質の異なる複数の利用画面があります。

## Chat

通常のClaude.aiのチャットです。

主な用途は次のとおりです。

* 質問と回答
* 文書作成
* 要約
* 翻訳
* 分析
* アイデア整理
* Web検索
* リサーチ
* ファイル分析
* 資料作成

利用者はWebブラウザだけでなく、デスクトップアプリやモバイルアプリからも利用できます。現在のEnterpriseプランでは、Web、デスクトップ、モバイルへのアクセスがシートに含まれます。 ([Claudeヘルプセンター][4])

## Claude Code

Claude Codeは、開発者向けのコーディングエージェントです。

単なるコード生成チャットではなく、利用者の許可と設定に基づいて、次のような処理を行います。

* コードベースを読む
* 複数のファイルを編集する
* コマンドを実行する
* テストを実行する
* Gitや開発ツールと連携する
* 不具合修正やリファクタリングを進める

ターミナル、IDE、デスクトップアプリ、Webなどで利用できます。Enterprise管理者は、Claude Codeが使用できるコマンド、ツール、MCPサーバー、接続先ドメインなどをManaged Settingsで制御できます。 ([Claude][5])

## Claude Cowork

Claude Coworkは、一般的な知識業務を複数ステップに分けて実行するためのエージェント機能です。

例えば、次のような作業をまとめて任せることを想定しています。

* 複数の資料を読み込む
* 社内システムから情報を集める
* 調査結果を整理する
* 文書やスプレッドシートを作成する
* ファイルを分類・整理する
* 定期的なタスクを実行する

Chatが「会話を1往復ずつ進める」利用方法であるのに対し、Coworkは「目的を渡して、複数段階の処理を進めてもらう」利用方法に近いものです。コネクター、ローカルファイル、プラグインなどを組み合わせて利用します。 ([Claude][6])

### 3つの違い

| 製品面         | 主な対象  | 基本的な使い方             |
| ----------- | ----- | ------------------- |
| Chat        | 全従業員  | Claudeと対話しながら仕事を進める |
| Claude Code | 開発者   | コードや開発環境に対して作業する    |
| Cowork      | 知識労働者 | 複数ステップの業務を委任する      |

Anthropicは、この3つをClaude Enterprise内の異なる利用面として位置付けています。 ([Anthropic][1])

# 5. チャット、プロジェクト、ファイルの関係

Claude.aiのデータ構造は、次のように理解できます。

```text
Claude.ai組織
│
├─ 単独のチャット
│   ├─ ユーザーの入力
│   ├─ Claudeの回答
│   ├─ 添付ファイル
│   └─ Artifacts
│
└─ プロジェクト
    ├─ プロジェクト指示
    ├─ ナレッジ／参照ファイル
    ├─ プロジェクト内チャットA
    ├─ プロジェクト内チャットB
    └─ 各チャットのArtifacts
```

## チャット

Claudeとの1つの会話単位です。

通常のチャットでは、会話中にファイルを添付し、Claudeに内容を分析させることができます。

## プロジェクト

プロジェクトは、特定の案件や業務ごとに情報をまとめる継続的な作業領域です。

例えば次のように利用します。

```text
プロジェクト名：新製品Aの企画
├─ 製品要件書
├─ 市場調査資料
├─ 過去の会議議事録
├─ Claudeへの共通指示
├─ 市場分析チャット
├─ 競合分析チャット
└─ 経営会議資料作成チャット
```

プロジェクト内に登録した指示やナレッジを、複数のチャットから利用できます。共有設定が許可されている場合は、プロジェクトを組織内で共有できます。 ([Claude][7])

## Artifacts

Artifactsは、Claudeが会話の横に生成する独立した成果物です。

例として、次のようなものがあります。

* 文書
* コード
* HTML
* 図表
* 分析結果
* インタラクティブなコンテンツ
* Webアプリの試作品

Artifactsはチャットの一部として保存されます。Enterpriseのデータ保持期間が満了してチャットが削除される場合、そのチャット内のArtifactsも削除対象になります。 ([Claudeヘルプセンター][8])

# 6. Connectorsによる外部システム連携

Claude Enterpriseでは、Claudeを社内のデータソースや業務システムへ接続できます。

公式に例示されている対象には、次のようなものがあります。

* Google Drive
* Gmail
* Google Calendar
* Microsoft 365
* Slack
* GitHub
* Linear
* その他のMCP対応サービス
* 自社で作成したカスタムコネクター

コネクターは、単にファイルをClaudeへコピーする機能ではありません。接続先から情報を検索・取得したり、設定によっては操作を実行したりできます。 ([Claudeヘルプセンター][9])

## アクセス権の考え方

コネクターでは、原則として次の2段階の権限制御が働きます。

```text
Claude側の許可
        AND
接続先システム側のユーザー権限
```

例えば、利用者がGoogle Drive上の特定ファイルを閲覧できなければ、Claudeもその利用者のコネクター経由ではファイルを取得できません。

管理者がコネクターを組織で有効にしても、それだけで全社員にデータアクセス権が付与されるわけではありません。通常、各利用者は接続先に対して個別に認証します。Enterprise-managed authを使用する構成もありますが、公式資料上では一部がベータ機能です。 ([Claudeヘルプセンター][9])

## 読み取りと書き込みの制御

管理者はコネクターの操作を、例えば次のように制御できます。

* 常に許可
* 実行前にユーザー承認が必要
* ブロック

したがって、次のようなポリシーが可能です。

```text
Slack：検索・読み取りは許可
Slack：メッセージ送信は承認必須

Google Drive：ファイル参照は許可
Google Drive：ファイル作成・編集は禁止
```

Claude側の設定は、接続先システムの権限を拡張するものではなく、接続先で許可されている操作をさらに狭める方向に働きます。 ([Claudeヘルプセンター][9])

# 7. メンバー、グループ、ロール

Claude Enterpriseのアクセス制御は、次の4要素で構成されます。

```text
ユーザー
  └─ グループに所属
       └─ グループにカスタムロールを割当
             └─ 利用できる機能・管理権限を決定
```

## 標準ロール

Claude.ai組織には、主に次の標準ロールがあります。

| ロール           | 概要                      |
| ------------- | ----------------------- |
| Primary Owner | 組織の最上位責任者。1組織につき1人      |
| Owner         | 組織設定、セキュリティ、請求等を管理      |
| Admin         | メンバー管理や一部の管理操作          |
| User          | 通常のClaude利用者            |
| Custom        | グループに割り当てられたカスタム権限のみを持つ |

Primary Ownerは1組織につき1つだけです。個人ではなくサービスアカウントをPrimary Ownerにする構成も公式に認められています。 ([Claudeヘルプセンター][10])

## グループ

グループは、組織内のユーザーを次のような単位でまとめる機能です。

* 開発部
* 法務部
* 人事部
* Claude Code利用者
* 管理職
* 高機密情報取扱者
* パイロット利用者

グループはClaude上で手動作成することも、SCIMを通じてIdPから同期することもできます。 ([Claudeヘルプセンター][11])

## カスタムロール

Enterpriseでは、部署ごとに利用できる機能を細かく分けられます。

例：

| カスタムロール   | 許可する機能                       |
| --------- | ---------------------------- |
| 一般社員      | Chat、Projects、Web検索          |
| 開発者       | Chat、Claude Code、コード実行       |
| リサーチ部門    | Chat、Research、Web検索、Projects |
| Cowork利用者 | Chat、Cowork、指定コネクター          |
| 請求担当      | 請求・利用量の閲覧                    |
| ID管理者     | SSO、SCIM、グループ、ロール管理          |

カスタムロールは直接ユーザーに割り当てるのではなく、原則としてグループへ割り当てます。Customユーザーが複数グループに所属する場合、通常の機能権限は加算され、いずれかのロールで許可された機能を利用できます。 ([Claudeヘルプセンター][11])

## 組織設定が上限になる

権限は次のような階層になっています。

```text
組織全体で機能を有効化
          ↓
カスタムロールで利用を許可
          ↓
グループを通じてユーザーへ適用
```

組織全体で無効にされた機能を、カスタムロールだけで有効にすることはできません。組織設定が「上限」、ロール設定がその範囲内での許可となります。モデルアクセスについても、組織で無効にしたモデルをカスタムロールが再度有効にすることはできません。 ([Claudeヘルプセンター][12])

# 8. SSO、JIT、SCIM

## SSO

SSOは、会社のIDを使ってClaudeへログインさせる仕組みです。

公式資料では、Claude EnterpriseはSAML 2.0およびOIDCによるSSOをサポートしています。 ([Claude][7])

```text
社員
  ↓
Microsoft Entra ID／Okta等
  ↓ 認証成功
Claude Enterprise
```

## JITプロビジョニング

JITでは、IdP上でClaudeの利用を許可されたユーザーが、初めてSSOログインした時点でClaude組織に登録されます。

設定が比較的簡単ですが、ユーザーが実際にログインするまでアカウントが作られないため、事前の一括管理には限界があります。

## SCIMプロビジョニング

SCIMでは、IdPからClaudeへユーザーやグループを同期します。

例えば、IdP上で社員を「Claude-Developers」グループに追加すると、Claude側でも次の処理を自動化できます。

* Claude組織へユーザーを追加
* 適切なグループへ所属させる
* ロールを割り当てる
* シートを割り当てる
* 退職・異動時にアクセスを変更する

JITはTeamとEnterpriseで利用でき、SCIMはEnterpriseとConsole組織で利用できます。 ([Claudeヘルプセンター][13])

# 9. Organization settingsという管理層

管理者向けの独立した別サイトがあるというより、Claude Enterpriseの主要な管理機能は、Claude.ai内の**Organization settings**にまとめられています。

主要な設定ページは次のように整理できます。

| 設定領域                    | 主な内容                           |
| ----------------------- | ------------------------------ |
| Members                 | ユーザー、招待、ロール、シート                |
| Groups                  | 部署・チーム別のグループ                   |
| Roles                   | カスタムロール、機能権限、管理権限              |
| Models                  | 利用可能モデル、標準モデル、推論負荷             |
| Organization and access | SSO、SCIM、ドメイン、セッション            |
| Connectors              | 外部サービス、操作権限                    |
| Libraries               | 組織共有のSkills、Plugins、Connectors |
| Data and privacy        | 保持期間、監査出力、暗号鍵、共有制御             |
| Usage                   | 組織・グループ・個人の利用上限                |
| Billing                 | シート、支払方法、請求                    |
| Analytics               | 利用者数、機能利用、Claude Code利用状況      |

Enterpriseのカスタム管理権限では、Identity & Access、Billing、Analytics、Privacy、User Management、Libraries、Directory managementなどに管理権限を分けることができます。 ([Claudeヘルプセンター][14])

つまり、「全員をOwnerにしないと管理を委任できない」という構造ではありません。例えば、次のような分業が可能です。

```text
情報システム部
  └─ SSO、SCIM、グループ、ロール

経理・FinOps
  └─ 請求、利用量、予算

セキュリティ部
  └─ データ保持、監査、ネットワーク制御

AI推進部門
  └─ Skills、Connectors、利用分析
```

# 10. データとセキュリティの構造

## データの管理主体

Claude for Workのデータは、利用者個人だけではなく、契約している企業組織の管理対象です。

Primary Ownerは組織データのエクスポートを要求でき、企業は利用可能なClaude機能を制限できます。企業アカウントの利用条件は、Anthropicと企業間の契約によって管理されます。 ([Claudeヘルプセンター][3])

## データ保持

Enterprise管理者は、チャットとプロジェクトについて独自の保持期間を設定できます。

現在の公式仕様では、最低保持期間は30日です。設定しない場合は無期限保持がデフォルトです。

* チャット：最後のメッセージから保持期間を計算
* プロジェクト：最後の更新から保持期間を計算
* プロジェクトの保持設定は、その中のチャットにも優先
* 期限後のデータは完全削除され、復旧不可

保持期間を短縮すると、新しい期間をすでに超えているデータが設定保存時に削除される可能性があります。 ([Claudeヘルプセンター][8])

## 暗号化鍵

通常はAnthropicが提供する暗号化が使用されます。

対象となるEnterprise組織では、CMEKを利用して、次のクラウド上で自社管理の鍵を準備できます。

* AWS KMS
* Google Cloud KMS
* Azure Key Vault

Anthropicはその鍵を使用して、チャット、プロジェクト、ファイルなどの対象データを暗号化します。鍵の利用操作は、顧客側クラウドの監査ログにも記録されます。 ([Claudeヘルプセンター][15])

## US-only inference

対象となる使用量課金型Enterpriseでは、モデルによる推論処理を米国内に限定する設定があります。

ただし、これは次の意味ではありません。

* データ保存場所を米国だけに限定する機能ではない
* SlackやGoogle Driveなど第三者サービスの処理場所を制御する機能ではない
* Claude Platform/API側のデータ所在地設定を変更するものではない

あくまで、Claude Enterprise組織の**モデル推論処理**を米国内に限定する設定です。 ([Claudeヘルプセンター][16])

# 11. ネットワークセキュリティ

## IP allowlist

Enterpriseでは、指定したIPアドレスやCIDR範囲からのみClaudeへのアクセスを許可できます。

例えば、次の経路だけを許可できます。

```text
本社ネットワーク
VPNの出口IP
承認済み拠点ネットワーク
```

許可リストにないIPアドレスからの認証済みリクエストはブロックされます。 ([Claudeヘルプセンター][17])

## Tenant Restrictions

Tenant Restrictionsは、企業ネットワークから個人のClaudeアカウントや、未承認のClaude組織を利用させないための制御です。

```text
社員が社内ネットワークからClaudeへアクセス
                  ↓
Proxyが許可組織IDをHTTPヘッダーへ設定
                  ↓
Anthropicが組織IDを検証
                  ↓
承認済みのEnterprise組織だけアクセス可能
```

これはシャドーIT対策として重要です。

* SSO：正規の企業組織へ安全にログインさせる
* IP allowlist：許可ネットワークからだけアクセスさせる
* Tenant Restrictions：許可されたClaude組織だけを使わせる

という違いがあります。 ([Claudeヘルプセンター][18])

# 12. 監査と分析

Claude Enterpriseには、性質の異なる3種類の可視化・取得機能があります。

## 監査ログCSV

Claude.aiの次の場所から出力します。

```text
Organization settings
  → Data and Privacy
  → Export logs
```

過去180日間の操作イベントをCSVとして取得できます。

ただし、監査ログCSVにはチャットやプロジェクトの本文は含まれず、一意のIDが中心です。 ([Claudeヘルプセンター][19])

## Compliance API

Compliance APIでは、プログラムから次の情報へアクセスできます。

* Activity Feed
* ユーザー
* グループ
* ロール
* 組織設定
* Claude.aiのチャット
* ファイル
* プロジェクト

セキュリティ、監査、法務部門が、SIEM、DLP、eDiscoveryなどへ情報を連携する用途です。 ([Claude Platform][20])

## Analytics

Analyticsは監査ではなく、導入・利用状況を見るための機能です。

例えば、次のような情報を扱います。

* アクティブユーザー
* 利用頻度
* 機能の利用状況
* Claude Codeの利用状況
* 部門ごとの利用傾向
* コストや利用量

整理すると、次の違いです。

| 確認したいこと               | 使用する機能                   |
| --------------------- | ------------------------ |
| 誰がいつログインしたか           | 監査ログ／Compliance API      |
| 誰がファイルをアップロードしたか      | 監査ログ／Compliance API      |
| チャット内容を取得したい          | Compliance API／データエクスポート |
| 今月何人が利用したか            | Analytics                |
| Claude Codeの利用が増えているか | Analytics                |
| 利用費が予算内か              | Usage／Billing            |

# 13. 利用量・課金の構造

現在の標準的な使用量課金型Enterpriseでは、次の2つが分かれています。

```text
シート料金
  └─ ユーザーがEnterpriseへアクセスする権利

使用量料金
  └─ Chat、Claude Code、Coworkで消費したトークン等
```

つまり、シートを購入しただけで一定量のトークンが含まれる構造ではなく、シート料金とは別に実際の使用量がAPIレートで請求されます。

管理者は次の単位で支出上限を設定できます。

* 組織全体
* グループ
* 個人ユーザー

グループ上限と個人上限の両方がある場合は、個人上限が優先されます。 ([Claudeヘルプセンター][4])

ただし、既存のEnterprise顧客には、Standard／Premium、Chat／Chat＋Codeなどの旧シート体系が残っている可能性があります。実際の導入案件では、公開ページだけで判断せず、契約書と管理画面のシートタイプを確認する必要があります。 ([Claude][7])

# 14. Claude EnterpriseとClaude Platformの境界

ここは特に重要です。

| 項目    | Claude Enterprise                    | Claude Platform                     |
| ----- | ------------------------------------ | ----------------------------------- |
| 主な利用者 | 一般社員、開発者、管理者                         | システム開発者                             |
| 主な画面  | Claude.ai、Desktop、Mobile、Claude Code | Claude Console                      |
| 主な目的  | 社員がClaudeを業務利用する                     | 自社システムへClaudeを組み込む                  |
| 認証    | ユーザーアカウント、SSO                        | APIキー等                              |
| データ   | チャット、プロジェクト、ファイル                     | APIリクエスト、Workspace                  |
| 管理単位  | Claude.ai組織                          | Console組織、Workspace                 |
| 課金    | シート＋使用量等                             | API使用量                              |
| 監査    | Audit Logs、Compliance API            | Admin API、Compliance Activity Feed等 |

両者は同じAnthropicのサービスですが、**同じ管理画面・同じデータ領域ではありません**。

親組織を通じてSSOやID管理を共有できる場合はありますが、Claude.ai組織とConsole組織は別組織として扱われ、請求や利用量もそれぞれで管理されます。 ([Claudeヘルプセンター][2])

# 15. 実際の処理フロー

社員がClaude Enterpriseを利用するときの流れを単純化すると、次のようになります。

```text
1. 社員がClaudeへアクセス
        ↓
2. 企業のIdPでSSO認証
        ↓
3. SCIM／JIT情報を基にClaude.ai組織へ所属
        ↓
4. グループとロールから利用可能機能を判断
        ↓
5. Chat／Claude Code／Coworkを利用
        ↓
6. 必要に応じてProjects、Files、Connectorsを参照
        ↓
7. Claudeモデルが推論して回答・成果物を生成
        ↓
8. チャットや成果物を組織データとして保存
        ↓
9. 設定した保持期間に従って管理・削除
        ↓
10. 操作イベントを監査ログへ記録
        ↓
11. 利用量をAnalyticsおよびBillingへ集計
```

コネクターを利用する場合は、途中に次の処理が入ります。

```text
Claude
  ↓
コネクターの組織ポリシーを確認
  ↓
ユーザーの接続先権限を確認
  ↓
Slack／Drive／M365等から情報取得
  ↓
取得情報を使ってClaudeが回答
```

# 導入担当者向けの要点

Claude Enterpriseの構造を設計する際は、少なくとも次の項目を別々に検討する必要があります。

| 設計領域     | 決める内容                            |
| -------- | -------------------------------- |
| 組織構造     | Claude.ai組織を1つにするか、複数にするか        |
| 親組織      | Claude Console組織とID管理を共有するか      |
| 認証       | SAMLかOIDCか、SSOをいつ強制するか           |
| プロビジョニング | 手動、JIT、SCIMのどれを使うか               |
| 権限       | 標準ロールとカスタムロールの使い分け               |
| グループ     | 部署、職種、機能別にどう分けるか                 |
| 機能       | Chat、Code、Cowork、Research等の許可範囲  |
| モデル      | 利用可能モデルと推論負荷の制限                  |
| コネクター    | 接続先、読み取り、書き込み、承認方式               |
| データ      | チャット・プロジェクトの保持期間                 |
| ネットワーク   | IP allowlist、Tenant Restrictions |
| 監査       | CSVかCompliance APIか、SIEM連携するか    |
| 費用       | 組織・グループ・個人の支出上限                  |
| 運用       | Primary Owner、管理者、監査担当者の分離       |

## 一言でまとめると

Claude Enterpriseは、次の三層で覚えると理解しやすいです。

```text
① Parent Organization
   会社全体のID・SSOを管理する層

② Claude.ai Organization
   ユーザー、データ、権限、請求、監査を管理する層

③ Claude Apps
   Chat、Claude Code、Coworkなどを実際に利用する層
```

その周囲に、Connectors、Projects、Skills、Compliance API、Analytics、ネットワーク制御などが配置される構造です。

なお、今回説明したのは公式資料から確認できる**製品・テナント・管理上の論理構造**です。Anthropic内部の物理的なクラウド構成、ネットワークトポロジー、マイクロサービス構成、データベース構成などは、確認した公開公式資料では詳細が開示されておらず、**不明**です。
