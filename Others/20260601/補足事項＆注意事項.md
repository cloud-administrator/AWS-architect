# 補足事項＆注意事項.md

## 1. まず押さえること

- Snowflake側のMCPサーバーは、Snowflake-managed MCP server objectです。通常のサーバープロセスをSnowflake上に立てるわけではありません。
- Codex側はMCP clientです。`~/.codex/config.toml`または`codex mcp add --url ...`でSnowflake MCP endpointを登録します。
- 本番設定は会社PC、会社Snowflake、会社Microsoft 365 tenantで実施します。自宅PCでは実接続やsecret保存をしません。

## 2. 認証の注意

推奨はOAuthです。PATは短期検証か暫定用途に限定します。

OAuthで注意すること:

- Snowflake OAuth integrationはDynamic Client Registrationを前提にしないため、client ID/secretの扱いを会社PCで検証します。
- Codex CLIはStreamable HTTP MCP serverのOAuth loginに対応していますが、会社PCに入っているCodexのバージョンでclient secretをどう扱うかは検証が必要です。
- 直結できない場合は、会社管理の中継MCP/API gatewayでSnowflake OAuthを処理する構成を検討します。

PATで注意すること:

- role restrictionを必ず設定します。
- 有効期限を短くします。
- network policyを使います。
- token secretは作成時にしか表示されません。
- Markdown、Git、チケット、チャットに貼り付けません。

## 3. 権限の注意

MCP serverへの`USAGE`だけでは不十分です。toolが参照するSnowflake objectにも権限が必要です。

| Tool | 追加で必要な主な権限 |
| --- | --- |
| `CORTEX_ANALYST_MESSAGE` | Semantic Viewへの`SELECT` |
| `CORTEX_SEARCH_SERVICE_QUERY` | Cortex Search Serviceへの`USAGE` |
| `CORTEX_AGENT_RUN` | Cortex Agentへの`USAGE` |
| `GENERIC` | UDF/SPへの`USAGE`、Warehouseへの`USAGE` |
| `SYSTEM_EXECUTE_SQL` | 実行するSQLに必要な各object権限 |

Snowflake-managed MCP serverのOAuth sessionは、接続ユーザーのdefault role/default warehouseに強く依存します。`DEFAULT_ROLE`と`DEFAULT_WAREHOUSE`を必ず確認してください。

## 4. Public接続とPrivate接続の注意

Private接続:

- 会社PCからPrivate Link用hostnameがPrivate EndpointのIPへ解決される必要があります。
- DNS、Firewall、NSG、OCSP到達性を確認します。
- `SYSTEM$ALLOWLIST_PRIVATELINK()`とSnowCDで確認します。

Public接続:

- Microsoft 365 Copilot CoworkのようなSaaSから接続する場合はpublic Snowflake URLを使います。
- public endpointを使う場合はnetwork policy、OAuth、RBACで制御します。
- Microsoft側の接続元IP範囲は変わり得るため、固定リストを資料内に埋め込まない方が安全です。

`privatelink-only`の注意:

- Snowflake accountでprivatelink-onlyを有効化すると、Public SaaSからの接続と競合する可能性があります。
- Codex Private接続とCopilot Cowork Public接続を同じSnowflake accountで併用する場合は、privatelink-onlyを有効化する前に必ず設計判断が必要です。

## 5. tool設計の注意

- Public向けMCP serverには`SYSTEM_EXECUTE_SQL`を入れない方針を推奨します。
- Private向けでも`SYSTEM_EXECUTE_SQL`は読み取り専用roleに限定します。
- tool名、title、descriptionには、個人情報、機密名、内部コード名を書かないでください。Snowflake metadataとして残ります。
- 複数MCP serverを使う場合は、tool poisoning / tool shadowingを避けるため、接続元と公開toolを確認します。
- Cortex Agentが別MCP serverを呼び、さらに元のAgentに戻るような再帰構成は避けます。

## 6. トラブルシュート

| 症状 | 見る場所 | 対応 |
| --- | --- | --- |
| CodexでMCP serverが見えない | `codex mcp list`、`/mcp` | URL、config.toml、enabled、OAuth login状態を確認 |
| OAuth loginが失敗する | Codex CLI、Snowflake OAuth integration | redirect URI、client ID、client secret対応、callback portを確認 |
| tools/listは見えるがtool実行に失敗 | Snowflake権限 | MCP serverだけでなく、Semantic View/Search/Agent/Warehouse権限を確認 |
| Private接続できない | DNS、Firewall、Private Endpoint | `Resolve-DnsName`、`Test-NetConnection`、SnowCDを確認 |
| PATで失敗する | PAT設定、network policy | role restriction、期限、network policy、headerを確認 |
| Public Coworkから接続できない | M365 plugin、Snowflake network policy | plugin配布、OAuthPluginVault、public URL、許可IPを確認 |

## 7. 検証で残すべき記録

- `SHOW MCP SERVERS`と`DESCRIBE MCP SERVER`の結果
- Codexの`codex mcp list`結果
- Private DNSと443疎通確認結果
- 代表的なtool実行結果
- Snowflake query history / access historyでの実行者確認
- エラーが出た場合の時刻、ユーザー、role、MCP server名

実データ、個人情報、PAT、client secretは検証記録に含めないでください。
