まず前提として、ChatGPT Enterprise/Edu の「クレジット」は **Deep Research / Thinking models / Image Gen / Advanced Voice など“高度な機能”を使うための共通残高**で、**ワークスペース全員が同じプールから消費**します（Enterprise/Edu は per-seat の上限はありません）。([help.openai.com](https://help.openai.com/en/articles/11487671-flexible-pricing-for-the-enterprise-edu-and-team-plans))

- **クレジットを使い切ったときに“追加料金が出ないように止める”方法（=上限超過を防ぐ）**
  - Enterprise/Edu は、**共有クレジットが 0 になると高度な機能が一時停止**します（ただし Owner が overages を有効化したり、追加クレジットを購入すると再開）。([help.openai.com](https://help.openai.com/en/articles/11487671-flexible-pricing-for-the-enterprise-edu-and-team-plans))
  - 追加料金（overage）を絶対に出したくない場合は、Owner が `Settings > Billing` で **hard overage limit を 0** に設定します（0 にすると overages に入れない、と公式に明記されています）。([help.openai.com](https://help.openai.com/en/articles/11487671-flexible-pricing-for-the-enterprise-edu-and-team-plans))

- **一人が使いすぎないようにする方法（デフォルトは？／余りはどうなる？）**
  - デフォルトは、Enterprise/Edu は **共有プール**かつ **ユーザー個別の上限なし**、さらに Usage limits も **None（制限なし）**なので、放っておくと一部の人が多く使ってプールを早く減らせます。([help.openai.com](https://help.openai.com/en/articles/11487671-flexible-pricing-for-the-enterprise-edu-and-team-plans))
  - 対策は、Owner が `Workspace settings > Permissions & roles > Usage limits` で **ユーザー（ロール）ごとの週次上限**を設定します（例：ライトユーザー／パワーユーザーで分ける）。([help.openai.com](https://help.openai.com/en/articles/20001001-setting-usage-limits-for-custom-roles-in-chatgpt-enterprise))
    - **Admin alert**：使いすぎを“検知”（止めない）。([help.openai.com](https://help.openai.com/en/articles/20001001-setting-usage-limits-for-custom-roles-in-chatgpt-enterprise))
    - **Hard cap**：上限に達したらそのユーザーは **高度な機能を使えなくなる**（クレジットを使わない機能は継続可）。([help.openai.com](https://help.openai.com/en/articles/20001001-setting-usage-limits-for-custom-roles-in-chatgpt-enterprise))
  - 「クレジットが余っていたら、使いすぎた人に割り当てられる？」→ Enterprise/Edu は最初から **個人割当ではなく共通プール**なので、“余りの再配分”という概念ではなく、**使った人がそのままプールを消費**します。([help.openai.com](https://help.openai.com/en/articles/11487671-flexible-pricing-for-the-enterprise-edu-and-team-plans))

**前提整理（「APIを止める」の意味）**
- ChatGPT Enterprise（`chatgpt.com`）の管理設定と、OpenAI API Platform（`platform.openai.com` / `api.openai.com`）の管理設定は別物です。例えば、ChatGPT の IP allowlisting は ChatGPT/Compliance API 向けで、API Platform には適用されません。 ([help.openai.com](https://help.openai.com/en/articles/10128477-chatgpt-enterprise-edu-release-notes%23.webm?utm_source=openai))

## 1) 「API経由でChatGPTを使わせない」ためのOpenAI側設定手順（API Platform側）
社内で“OpenAI API（`api.openai.com`）を使ったChatGPT相当の推論”をさせたくない場合、基本は「ユーザーをAPI Platformのプロジェクトに入れない」「APIキーを無効化する」「権限でモデルリクエスト/キー管理を潰す」の3点です。

- **手順A：API Platformのメンバー/プロジェクト付与を見直す**
  - API Platformでは **プロジェクトに所属すると推論（inference）やキー作成が可能** です（例：OrgのReader、ProjectのMemberはいずれも “perform inference / create keys”）。 ([help.openai.com](https://help.openai.com/en/articles/9186755-managing-your-work-in-the-api-platform-with-projects%23.csv))
  - なので「APIを使わせない対象ユーザー」は、**API Platform組織や各プロジェクトのメンバーから外す**のが最短です（Project Owner が Members から Remove 可能）。 ([help.openai.com](https://help.openai.com/en/articles/9186755-managing-your-work-in-the-api-platform-with-projects%23.csv))

- **手順B：既存のAPIキーを削除（失効）する**
  - UIでの削除：API keyページで対象キーの**ゴミ箱アイコン**から削除（削除したキーは利用不可）。 ([help.openai.com](https://help.openai.com/en/articles/9047852?utm_source=openai))
  - どこで管理するか：組織設定でプロジェクトを選び **API Keys** から作成/編集できます。 ([help.openai.com](https://help.openai.com/en/articles/9186755-managing-your-work-in-the-api-platform-with-projects%23.csv))

- **手順C：RBACで「モデルを呼べない／キーを作れない」状態を作る（必要に応じて）**
  - API PlatformのRBACは、権限をロールで細かく制御でき、**APIとDashboardの両方に同じ権限が効く**設計です。 ([platform.openai.com](https://platform.openai.com/docs/guides/rbac?utm_source=openai))
  - “APIを使わせない”目的なら、少なくとも **Model capabilities（Request）** や **Project API Keys** を一般ユーザーに付与しない方針にします。 ([platform.openai.com](https://platform.openai.com/docs/guides/rbac))
  - 反映には最大30分かかることがあるため、非Ownerアカウントで動作確認する流れが推奨されています。 ([platform.openai.com](https://platform.openai.com/docs/guides/rbac))

- **（強化策）mTLSで「社内証明書がないとAPI呼び出し不可」にする**
  - OpenAI Mutual TLS を構成すると、APIリクエストは `https://mtls.api.openai.com` 宛てにし、**正しいAPIキー＋クライアント証明書が揃わないと受け付けない**形にできます（※ベータ）。 ([help.openai.com](https://help.openai.com/ja-jp/articles/10876024-openai-%E7%9B%B8%E4%BA%92tls-%E3%83%99%E3%83%BC%E3%82%BF%E3%83%97%E3%83%AD%E3%82%B0%E3%83%A9%E3%83%A0?utm_source=openai))

## 2) APIを止めたら、Codex（VS Code/CLI）は使える？
結論：**「OpenAI API（`api.openai.com`）を使わせない」だけなら、Codexは“ChatGPTサインイン方式”で継続利用できる可能性が高い**です。理由は、Codexが2通りの認証/経路を持つためです。

- **Codexの認証方式は2つ**
  - ChatGPTサインイン（サブスク枠） / APIキー（従量課金枠）。 ([developers.openai.com](https://developers.openai.com/codex/auth))
  - さらにCodex CLIは、**ChatGPTログイン時は `chatgpt.com` のエンドポイント**、**APIキー時は `api.openai.com` のエンドポイント**を使います。 ([openai.com](https://openai.com/fil-PH/index/unrolling-the-codex-agent-loop/))

- **つまり**
  - `api.openai.com` 側（API Platform）を無効化しても、**ChatGPTログインを許可している限り**、Codex（CLI/IDE拡張）が動く構成は取り得ます。 ([developers.openai.com](https://developers.openai.com/codex/auth))
  - ただし、企業側で **Codex Local を許可していない**とCLI/IDEは使えません。Workspace Settings で **“Allow members to use Codex Local”** をオンにする必要があります（オフだと403エラー）。 ([developers.openai.com](https://developers.openai.com/codex/enterprise))
  - VS Codeについては、公式に「Codex VS Code extensionがVS Codeフォークでも動く」旨が記載されています。 ([help.openai.com](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan/))

- **「Codex経由でAPIキーを使われる」ことも防ぎたい場合**
  - 端末側の管理設定で、Codexのログイン方式を **`forced_login_method = "chatgpt"`** に固定できます（APIキー認証を許可しない）。必要なら **`forced_chatgpt_workspace_id`** で接続先ワークスペースも固定できます。 ([developers.openai.com](https://developers.openai.com/codex/auth))
  - ChatGPTワークスペース側では、RBACで **Codexの利用可否** をロール単位で制御できます。 ([help.openai.com](https://help.openai.com/en/articles/11750701?utm_source=openai))

## （補足）「ChatGPTが外部APIを呼べるのを止めたい」場合（GPT Actions）
もし“API経由”が「GPT Actionsで外部APIへ接続できること」を指しているなら、別の設定です。
- 管理画面の GPT Actions ドメイン設定で、**Allow all domainsをオフ＋許可ドメイン一覧を空**にすると、ワークスペースでGPT Actionsをブロックできます。 ([help.openai.com](https://help.openai.com/en/articles/9442513-gpt-actions-domain-settings-chatgpt-enterprise%23.apk?utm_source=openai))

必要なら、御社の要件が「(A) OpenAI API Platform禁止」なのか「(B) GPT Actions禁止」なのか「(C) 両方」なのかを前提に、運用設計（どの部署だけ例外でAPI可にする等）まで含めて手順を整理します。

ChatGPT Enterprise で CSV から一括招待する場合、OpenAI 公式ヘルプでは **CSV は「email」と「role」の2列**で、**1行目にヘッダー行**を置く形式になっています。 ([help.openai.com](https://help.openai.com/en/articles/8266401-how-do-i-add-change-or-remove-chatgpt-enterprise-members))

**CSVに書く内容（必須）**
- 1列目: `email`（招待する人のメールアドレス）
- 2列目: `role`（付与する権限。例では `member` / `admin` / `owner`） ([help.openai.com](https://help.openai.com/en/articles/8266401-how-do-i-add-change-or-remove-chatgpt-enterprise-members))

**CSV例（そのまま使える形）** ([help.openai.com](https://help.openai.com/en/articles/8266401-how-do-i-add-change-or-remove-chatgpt-enterprise-members))
```csv
email,role
user1@company.com,member
admin@company.com,admin
it@company.com,owner
```

**権限まわりの注意（重要）**
- `Owner` は `Admin/Owner/Member` を招待・削除・ロール変更できます。 ([help.openai.com](https://help.openai.com/en/articles/8266401-how-do-i-add-change-or-remove-chatgpt-enterprise-members))
- `Admin` は `Member` だけを招待・削除できます（他ロールは扱えません）。 ([help.openai.com](https://help.openai.com/en/articles/8266401-how-do-i-add-change-or-remove-chatgpt-enterprise-members))
- `Admin` が `owner` や `admin` を含む CSV をアップロードすると、許可されないロールは `Member` 扱いに変更される旨のメッセージが表示されます。 ([help.openai.com](https://help.openai.com/en/articles/8266401-how-do-i-add-change-or-remove-chatgpt-enterprise-members))

**どこからアップロードするか**
- ChatGPT の「Manage workspace settings」内の「Members」タブから、CSV をアップロードして一括追加します。 ([help.openai.com](https://help.openai.com/en/articles/8266401-how-do-i-add-change-or-remove-chatgpt-enterprise-members))

**結論（Entra ID SSO＋「APIキー禁止／ブラウザログイン必須」）**
- Entra ID（SAML）SSO の ChatGPT Enterprise 環境でも、Codex を **VS Code 拡張**／**Codex CLI** で利用できます。OpenAI の SSO は ChatGPT で SAML をサポートしており、一般的な IdP として Azure AD / Entra ID が挙げられています。 ([help.openai.com](https://help.openai.com/en/articles/10468051-sso-overview))  
- Codex の「Sign in with ChatGPT」は **ブラウザを開いてログイン**し、ログイン後にトークンが CLI/拡張へ戻る方式です（= “必ずWebブラウザ経由でログイン” の要件に合致）。 ([developers.openai.com](https://developers.openai.com/codex/auth))  
- Codex CLI/IDE 拡張は APIキー認証も可能ですが、**管理者側で ChatGPT ログインだけに固定**できます（`forced_login_method = "chatgpt"`。条件不一致ならログアウトして終了）。 ([developers.openai.com](https://developers.openai.com/codex/auth))  

**前提：Enterprise 側の「Codex Local」許可が必要**
- ChatGPT Enterprise の管理画面で **“Allow members to use Codex Local”** を ON にしないと、CLI/IDE から使おうとしても `403 - Unauthorized` になります。 ([help.openai.com](https://help.openai.com/en/articles/11390924))  

**「API経由推論を禁止」の解釈と、Codex での実現方法（重要）**
- Codex は認証方式が2つあります：  
  - `Sign in with ChatGPT`（サブスクリプション利用）  
  - `API key`（OpenAI Platform の通常API課金） ([developers.openai.com](https://developers.openai.com/codex/auth))  
- 「APIキー経由を禁止したい」なら、`forced_login_method = "chatgpt"` を配布して **APIキーでの利用を技術的に封じる**のが公式に案内されている方法です。 ([developers.openai.com](https://developers.openai.com/codex/auth))  
- なお、ChatGPT ログインを使うと Codex の扱い（権限/保持/レジデンシ等）は **ChatGPT Enterprise 側のポリシー**に従い、APIキー利用時は **API組織側の設定**に従う、と明記されています（どちらの統制下で動かしたいかの判断ポイント）。 ([developers.openai.com](https://developers.openai.com/codex/auth))  

**SSO（Entra）運用で詰まりやすい点：CLI のブラウザ戻りが塞がれるケース**
- CLI は通常「ブラウザでログイン→localhost へのコールバック」でトークンを受け取りますが、企業ネットワークやリモート/ヘッドレス環境で **localhost コールバックがブロック**されることがあります。 ([developers.openai.com](https://developers.openai.com/codex/auth))  
- その場合は **Device Code 認証（beta）**を使えます。管理者がワークスペース権限で有効化し、CLI では `codex login --device-auth` を使う方式です（これもブラウザでサインインします）。 ([help.openai.com](https://help.openai.com/en/articles/11390924))  

必要なら、貴社の要件（「API禁止」が “APIキー禁止” なのか “OpenAIへの外部通信自体を禁止” なのか）を前提に、Codex Local を許可する/しないの判断材料を整理して文章化します。
